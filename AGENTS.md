## 6. Product rules that must never be violated

### Truthfulness

- AI may improve wording, order, clarity, and relevance.
- AI must never invent skills, projects, experience, education, certifications, achievements, or results.
- Every factual resume claim must be traceable to trusted profile data or confirmed evidence.
- Missing skills remain missing.
- Learning skills may appear only as learning, never as professional experience.
- AI-originated profile changes remain proposals until explicitly approved.
- Approved resume versions are immutable.

### Explainability

- Match scores must be deterministic Laravel calculations.
- LLM output may assist extraction, normalization, classification, and explanation.
- LLM output must not directly determine the final numeric match score.
- Store provenance, confidence, prompt/model version, profile version, job-analysis version, and scoring version when relevant.
- Critical missing requirements must be shown independently from the weighted score.

### Ownership and privacy

- Candidate-owned records are private by default.
- Every resource lookup requires ownership scoping and policy authorization.
- Route model binding is not an authorization check.
- A user must never access another user's data by changing an identifier.
- Sensitive administrator access must have a documented reason and an audit event.
- Do not log raw CV text, passwords, cookies, tokens, API keys, or unnecessary PII.

## 7. Architecture standards

### Backend architecture

Use an API-first modular monolith.

Approved domains:

- Identity
- Profile
- Skills
- Opportunities
- Matching
- Resumes
- Applications
- Learning
- AI Operations
- Platform

Preferred organization:

```text
backend/app/
├── Domain/<Domain>/
│   ├── Actions/
│   ├── Data/
│   ├── Enums/
│   ├── Events/
│   ├── Policies/
│   └── Services/
├── Http/
│   ├── Controllers/Api/V1/
│   ├── Requests/Api/V1/
│   └── Resources/Api/V1/
└── Support/
    ├── Audit/
    ├── Idempotency/
    ├── Pagination/
    └── ProblemDetails/
```

Rules:

- Controllers remain thin.
- Use Form Requests for HTTP validation.
- Use Policies for authorization.
- Use typed immutable DTOs at application/domain boundaries.
- Domain actions or services own workflows and transaction boundaries.
- Eloquent models own relationships and small invariants, not complete workflows.
- Use API Resources; never return models directly.
- Dispatch queued jobs after database commit.
- Prevent N+1 queries.
- Paginate collections.
- Use stable domain exceptions mapped to safe API problem codes.
- Application code reads environment values through configuration, not direct `env()` calls.
- Do not add a package when Laravel already provides an appropriate capability.
- Do not add or change a dependency without approval in the active change.

Use:

- Vue 3 Composition API
- TypeScript strict mode
- `<script setup lang="ts">`

Rules:

- Keep routes, queries, mutations, schemas, pages, and components in the owning feature.

- Use semantic HTML, keyboard support, visible focus, labels, and accessible announcements.
- Never render unsanitized `v-html`.
- Handle HTTP 401 and 419 centrally without infinite retry loops.
- Lazy-load routes and avoid unnecessary large dependencies.

## 8. API standards

- Base path: `/api/v1`.
- Format: JSON UTF-8.
- Dates: ISO 8601 UTC.
- Authentication: Laravel Sanctum first-party SPA cookie authentication.
- CSRF protection is mandatory.
- Never store authentication tokens in browser localStorage.
- Use Eloquent API Resources.
- Accept or generate `X-Request-ID`.
- Return and propagate request IDs through logs, jobs, provider calls, and exports.
- Async creation endpoints return `202 Accepted` with an operation resource and polling endpoint.
- Use cursor pagination for feeds/history.
- Use allowlisted filters and sort fields.
- Use optimistic concurrency for editable profile and resume resources.
- Maintain OpenAPI 3.1 and keep it synchronized with implementation.

API errors use RFC 9457-style problem details:

```json
{
  "type": "https://careerpilot.example/problems/example",
  "title": "Readable title",
  "status": 422,
  "detail": "Readable detail",
  "instance": "/api/v1/example",
  "code": "stable_problem_code",
  "errors": {},
  "request_id": "req_..."
}
```

Never expose stack traces or internal exception messages to clients.

## 9. Database standards

- Use MySQL with InnoDB, utf8mb4, foreign keys, and strict SQL mode.
- Prefer normalized relational tables.
- Use JSON only for bounded, versioned, flexible data that is not heavily queried.
- Index every foreign key.
- Add indexes for frequent filters and sorts.
- Use unique constraints to enforce invariants.
- Verify query plans with `EXPLAIN` when performance matters.
- Store timestamps in UTC.
- Prefer PHP enums stored as strings over MySQL `ENUM`.
- Use soft deletes only when recovery or audit value justifies them.
- Every migration must include rollback or documented forward-fix reasoning.
- Do not delete or reset existing user databases during normal development work.
- Inspect the current schema before creating migrations or models.

Important invariants include:

- One active candidate profile per user.
- One candidate-skill row per profile and normalized skill.
- One application per user and opportunity.
- One answer per clarification question.
- Immutable approved resume versions.
- Match snapshots reference exact source versions.
- Ownership foreign keys and indexes are mandatory.

## 11. AI engineering rules

- OpenAI API calls are server-side only.
- Use Laravel AI SDK.
- Preserve a provider abstraction.
- Use structured outputs or strict schema validation.
- Validate all AI output in Laravel before persistence.
- Treat CVs, job descriptions, company pages, and uploaded files as untrusted data, not instructions.
- Delimit untrusted content in prompts.
- Give agents minimum tool access.
- Never give a research agent unrestricted shell, database-write, or network access.
- Record provider, model, prompt version, response identifier when available, latency, tokens, estimated cost, and status.
- Apply quotas, concurrency controls, timeout, retry, and cancellation.
- Default automated tests use fakes.
- Live provider tests are opt-in, isolated, budget-limited, and excluded from normal CI.
- Never weaken truthfulness or safety during provider fallback.
- Never claim guaranteed ATS performance or hiring outcomes.

AI activation gates:

- Structured schema validity: at least 99%.
- Critical factual resume hallucinations: zero tolerated.
- Required/preferred skill classification accuracy: at least 90%.
- In-domain skill normalization precision: at least 90%.
- Useful clarification rate: at least 80%.
- Duplicate clarification questions: below 5%.
- Provider failure must produce a safe status with no partial trusted update.
